# 주니어 개발자 온보딩이 내부 고객의 개발 병목을 줄인다

> Track: [[Track_6_Revenue]] | ID: `hyp-6-01` | 상태: planning

## 가설

**Q: 주니어 개발자 온보딩이 내부 고객(기획/운영/대표)의 개발 요청 병목을 줄여 전체 개발 리드타임을 단축하는가?**

---

## 판정 기준

### 성공 (Success)
온보딩 완료(첫 PR merge) 이후 4주 기준,
- 내부 요청 태스크 **중앙값 리드타임(To Do→Done)**: **25% 이상 감소**
- 내부 요청 태스크 **Blocked(hold/blocked) 체류일수 평균**: **20% 이상 감소**
- 내부 요청 태스크 **주간 Done 처리량**: **15% 이상 증가**

### 실패 (Failure)
온보딩 완료 이후 4주 기준,
- 내부 요청 태스크 리드타임 개선 **5% 미만**
- 또는 Blocked 체류일수 평균이 **10% 이상 증가**
- 또는 주간 Done 처리량이 **10% 이상 감소**

---

## 측정 방법

**데이터 소스**: Dashboard/Kanban 태스크 타임스탬프(To Do/Doing/Hold/Done) + 태스크 타입/태그(내부요청) + PR 메트릭(리뷰 대기시간/머지까지 시간) + 주간 리포트(병목 사유 라벨)

**측정 지표**:
- 내부 요청 태스크 리드타임(중앙값/분포): To Do 시작일 → Done 완료일
- 상태별 체류시간: Doing 체류, Hold/Blocked 체류(일)
- 주간 처리량: 내부 요청 Done 개수/주
- PR 메트릭(보조): PR 생성→머지 시간, 리뷰 대기시간, 리뷰 코멘트 라운드 수
- 병목 사유 Top3(정성): Blocked 라벨/코멘트 분류(의존성/요구사항 불명확/환경/리뷰 병목)

## 만약 실패한다면?

### 영향
- 인력 투입이 곧바로 병목 해소로 이어지지 않아, 내부 요청 처리속도/제품 개발속도 개선이 제한됨
- 리뷰/의사결정/요구사항 품질 등 “비개발 병목”이 핵심 원인일 가능성

### 대응 (피벗/중단)
- 피벗 1: 내부 요청 템플릿/요구사항 명세(AC) 표준화 + triage 미팅 도입으로 병목 원인 제거
- 피벗 2: 코드리뷰 SLA/리뷰어 로테이션 도입(리뷰 병목이면) + 작은 PR 가이드 적용
- 피벗 3: 주니어 업무범위를 “내부 자동화/버그픽스 전담”처럼 명확히 재설계(책임 경계 강화)
- 중단: 2회(각 4주) 반복 측정에서도 개선이 5% 미만이면, 온보딩이 아니라 “프로세스/우선순위 시스템”을 1차 레버로 전환
